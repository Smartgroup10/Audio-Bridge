RNF-V01 Integracion Voz Recepcion de llamadas entrantes desviadas desde Centralita de Notaria via red publica La solucion debe permitir la recepcion de llamadas via red publica pudiendo diferenciar que llamadas corresponden a cada Notaria M Diseno sistema Asterisk 18+ res_audiosocket TCP :9092 Go AudioSocket
Formato E.164. Routing por DDI via dialplan (CALLERID(dnid)). Resolucion de tenant via TenantRegistry.LookupByDDI(). Cada notaria registra sus DDIs en config.yaml. AudioSocket listener en TCP :9092, protocolo binario (UUID + PCM frames).
RNF-V02 Integracion Voz Recepcion de llamadas entrantes desviadas desde Centralita de Notaria via trunk SIP La solucion debe permitir la recepcion de llamadas via SIP trunk pudiendo diferenciar que llamadas corresponden a cada Notaria M Diseno sistema PJSIP TLS/SRTP SIP Trunk por Notaria
Trunk SIP dedicado por tenant (sip_trunk: "trunk-notaria-N001"). Securizacion via TLS/SRTP en pjsip.conf. DDI como identificador de tenant en el dialplan. Soporta multiples trunks simultaneos con max_concurrent: 50.
RNF-V03 Integracion Voz Traspaso de audio al modulo externo de Inteligencia Artificial La solucion debe de ser capaz de enviar el stream de audio al modulo de IA de forma bidireccional M Diseno sistema AudioSocket TCP WebSocket TLS (WSS) OpenAI Realtime API G.711 mu-law
1 stream bidireccional por llamada. Asterisk → AudioSocket (PCM 16-bit signed LE, 16kHz, mono) → Bridge (Go) → WSS (G.711 mu-law, 8kHz) → OpenAI Realtime. Conversion de codec en openai_adapter.go (encode/decode mu-law). Jitter buffer con pre-buffering 60ms (audioBuffer). Playback a tasa constante 20ms/frame.
RNF-V04 Integracion Voz Traspaso de informacion contextual al modulo externo de Inteligencia Artificial La solucion debe tener la capacidad de facilitar al modulo de IA informacion contextual relacionada con la llamada M Diseno sistema WSS Query Params session.update ConnectParams
Parametros en handshake WSS: notaria_id, caller_id, interaction_id (UUID), call_type (inbound/callback), schedule (business_hours/after_hours), ddi_origin, context_id, context_data (expediente, motivo). Session update con instrucciones personalizadas por notaria via session.update de OpenAI.
RNF-V05 Integracion Voz Transferencia de llamadas via red publica La solucion debe ser capaz de transferir llamadas a personas especificas de Notaria en caso de no resolucion del Asistente Virtual via red publica M Diseno sistema AMI Redirect Function Calling Extensiones + Group Hunt
Personas, extensiones y grupos de salto valorable. AMI Action Redirect con destino configurable por tenant: transfers.default (ej: "201"), transfers.extensions (recepcion, notario, oficial), transfers.group_hunt (ej: "300"). Fallback automatico si IA no disponible (fallbackTransfer()). IA solicita transfer via function calling: transfer_call.
RNF-V06 Integracion Voz Transferencia de llamadas via trunk SIP La solucion debe ser capaz de transferir llamadas a personas especificas de Notaria en caso de no resolucion del Asistente Virtual via trunk SIP M Diseno sistema AMI Redirect SIP Trunk por Tenant transfer_call
Personas, extensiones, grupos de salto. AMI Action Redirect con contexto from-internal + trunk SIP del tenant (sip_trunk). OpenAI function calling: transfer_call con destination, destination_type (extension/external), via: "sip_trunk". Evento de retorno: transfer_completed con status connected/failed.
RNF-V07 Integracion Voz Securizacion de las comunicaciones con el modulo externo de Inteligencia Artificial La interconexion con el modulo de inteligencia artificial debe estar securizada M Diseno sistema WSS (TLS 1.2+) API Key Bearer Token Bcrypt Rate Limiting
WSS, API Key, Bearer Token. Conexion IA: WebSocket sobre TLS (:443) con header X-API-Key. Admin API: Token Bearer con expiracion 24h. Password admin: soporte password_hash bcrypt (golang.org/x/crypto/bcrypt). Rate limiting per-IP (token bucket: 30 burst, 1/s refill). CORS middleware restrictivo.
RNF-V08 Integracion Voz Logs y trazas El sistema debe disponer de logs, registro de llamadas, eventos, streams, interacciones y calidad de llamada (MOS) que permita tracear end-to-end las llamadas S Diseno sistema Zap Logger (JSON) SQLite WAL UUID Correlation WAV Dual Recording
Logging estructurado con Zap (formato JSON). SQLite tabla interaction_logs: call_id, timestamp, direction, content, event_type. call_id (UUID v4) como correlation ID end-to-end. Tipos de evento: speech, transfer, hangup, DTMF. Grabacion dual WAV (canal llamante + canal IA) con recording.Recorder. Transcripciones en tiempo real almacenadas en DB. MOS: pendiente de implementar.
RNF-V09 Integracion Voz Reporting asociado La solucion debe disponer de informes de trafico entrante/saliente con un minimo de datos para poder analizar y supervisar el funcionamiento sin necesidad de descargar y tratar los logs y trazas M Diseno sistema REST API /admin/stats Vue 3 Dashboard Pinia Store SSE Streaming
Timestamp, ID interaccion, llamante, llamado, resultado, duracion. API REST: /api/v1/admin/stats (KPIs), /api/v1/admin/calls (historial paginado con filtros). Panel Vue 3: Dashboard con llamadas hoy, activas, duracion media, desglose por notaria (inbound/outbound). Historial con filtros: notaria, rango fechas, estado. Paginacion server-side (20/pagina). Detalle de llamada: transcripcion completa, logs de interaccion, metadatos, grabaciones.
RNF-V10 Integracion Voz Logs y trazas El sistema debe disponer de logs, registro de llamadas, eventos, streams, interacciones y calidad de llamada (MOS) que permita tracear end-to-end las llamadas S Diseno sistema SSE (Server-Sent Events) LiveTranscript SQLite interaction_logs SSEHub Broadcast
Trazabilidad en tiempo real via SSE (/api/v1/admin/sessions/live). Eventos: call_started, transcript_user, transcript_ai, call_ended. Panel: componente LiveTranscript.vue con scroll automatico y mensajes timestamped. Indicador de conexion SSE en sidebar (connected/connecting/disconnected). Almacenamiento persistente en SQLite para consulta posterior. MOS: pendiente de implementar (requiere analisis de jitter/packet loss).
RNF-V11 Integracion Voz Emision de llamadas automaticas La solucion debe permitir emitir llamadas salientes de forma automatizada y enlazarlas con el modulo de IA una vez el cliente haya descolgado M Diseno sistema AMI Originate REST API Call Progress Detection Retry con backoff
Integracion via API REST + AMI Originate. Endpoint: POST /api/v1/calls/outbound con payload: destination, notaria_id, call_type (callback), context_id, context_data (expediente, motivo). AMI Action Originate con variables: CALL_UUID, NOTARIA_ID, CALL_TYPE. Retry configurable: 3 intentos, 30s intervalo (OriginateWithRetry()). Trunk SIP del tenant como caller ID. Call Progress Analysis via Asterisk. Conexion automatica a IA post-descolgado via AudioSocket.
