1 Resumen Ejecutivo Que se hizo y por que

El Consejo de los Trabajadores Notariales (CTN) ha aceptado el presupuesto para integrar su IA de voz Lakimi con nuestro Audio Bridge. El modelo de integracion cambia respecto al actual:

Aspecto Modo actual (OpenAI) Modo CTN (Lakimi)
Conexion IA Bridge se conecta a OpenAI via WSS Lakimi se conecta al Bridge (somos servidor WSS)
Acciones (transfer/hangup) Llegan por el canal WSS como eventos Llegan por HTTP REST (endpoints nuevos)
Resolucion de tenant Config local YAML / DB SQLite Backoffice CTN via API HTTP + cache local

Estos cambios se han implementado sin romper nada del flujo actual. El modo OpenAI sigue funcionando exactamente igual. Los cambios preparan la infraestructura para que, cuando Lakimi entregue su spec, solo haya que implementar el conector WSS server.

Estado actual Todos los cambios estan desplegados y en produccion en pbxtest.smartgroup.es. El servicio audio-bridge esta activo y los endpoints existentes + nuevos responden correctamente.

3 Nuevos Endpoints HTTP API para que Lakimi controle las llamadas

Ambos endpoints requieren autenticacion via header X-API-Key o Authorization: Bearer {key}.

POST /api/v1/calls/:call_id/transfer

Descripcion

Transfiere una llamada activa a un destino (extension, numero externo, grupo). Lakimi llamara a este endpoint cuando su IA decida que debe transferir al usuario.

Request Body

{
  "destination": "200",              // Extension o numero destino
  "destination_type": "extension"    // Opcional: "extension", "external", "queue"
}

Respuestas

CodigoDescripcionBody
200 Transfer ejecutado {"status": "transferred", "call_id": "...", "destination": "200"}
400 Falta destination {"error": "..."}
404 Call ID no encontrado {"error": "call not found"}
500 Fallo AMI al transferir {"error": "AMI transfer failed: ..."}

Ejemplo cURL

curl -X POST http://pbxtest.smartgroup.es:8080/api/v1/calls/abc-123/transfer \
  -H "X-API-Key: changeme_api_key" \
  -H "Content-Type: application/json" \
  -d '{"destination": "200", "destination_type": "extension"}'
POST /api/v1/calls/:call_id/hangup

Descripcion

Cuelga una llamada activa. Lakimi llamara a este endpoint cuando su IA determine que la conversacion ha terminado.

Request Body

{
  "reason": "conversation_complete"   // Opcional, default: "api_hangup"
}

Respuestas

CodigoDescripcionBody
200 Hangup ejecutado {"status": "hung_up", "call_id": "...", "reason": "..."}
404 Call ID no encontrado {"error": "call not found"}

Ejemplo cURL

curl -X POST http://pbxtest.smartgroup.es:8080/api/v1/calls/abc-123/hangup \
  -H "X-API-Key: changeme_api_key" \
  -H "Content-Type: application/json" \
  -d '{"reason": "conversation_complete"}'
Endpoints existentes sin cambios POST /calls/precreate, POST /calls/outbound, GET /calls/:id/status, GET /calls/active, GET /stats — todos siguen funcionando exactamente igual. El panel admin tambien sigue intacto.
4 Metodos Publicos del Bridge Logica reutilizable para transfer y hangup

Se extrajeron dos metodos publicos del Bridge que encapsulan toda la logica de transferencia y colgado. Son invocados tanto por la IA (via WebSocket events) como por la API HTTP (via endpoints REST).

TransferCall

func (b *Bridge) TransferCall(callID, destination, destType string) error

Flujo interno:

  1. Busca la llamada en el CallRegistry por callID
  2. Cambia estado a CallStateTransfer y guarda el destino
  3. Inserta log de interaccion en la DB (event_type: "transfer")
  4. Broadcast SSE event call_transfer al panel admin
  5. Ejecuta AMI.Transfer(channel, destination, "from-users")
  6. Si OK → marca la llamada como completada con reason "transfer"
  7. Si error → devuelve error (el caller decide que hacer)

HangupCall

func (b *Bridge) HangupCall(callID, reason string) error

Flujo interno:

  1. Busca la llamada en el CallRegistry
  2. Si tiene canal Asterisk, ejecuta AMI.Hangup(channel)
  3. Marca la llamada como completada con el reason proporcionado
  4. Inserta log de interaccion en la DB (event_type: "hangup")
Doble invocacion Estos metodos ahora son llamados desde dos puntos: (1) processAIMessages cuando la IA envia un evento WebSocket, y (2) los handlers HTTP cuando Lakimi/CTN llama a los endpoints REST. La logica es identica en ambos casos.
5 Selector de Modo IA Como el bridge decide que adaptador de IA usar

En HandleAudioSocket, el codigo que decide como conectarse a la IA se ha refactorizado de un if/else a un switch con tres modos:

Modo Config YAML Comportamiento Estado
"openai" ai.type: "openai" Bridge se conecta a OpenAI Realtime API via WSS. Transcripcion por callbacks. Es el flujo actual para notarias. Produccion
"ctn" ai.type: "ctn" Modo CTN/Lakimi. Lakimi se conecta al Bridge como servidor WSS. Actualmente: placeholder que hace fallback transfer directo. Placeholder
default Cualquier otro valor WSS client generico (custom AI module). Conexion a ai.endpoint. Produccion 
switch b.cfg.AI.Type {
case "openai":
    // Nos conectamos a OpenAI Realtime (flujo actual)
    oaiClient := wssclient.NewOpenAIRealtimeClient(oaiCfg, logger)
    // ... setup transcript callbacks ...
    oaiClient.Connect(ctx, connectParams, oaiCfg)
    aiClient = oaiClient

case "ctn":
    // TODO: implementar WSS server para Lakimi
    logger.Warn("CTN mode: not yet implemented")
    b.fallbackTransfer(call, tenant, logger)
    return

default:
    // WSS client generico
    wssClient := wssclient.NewClient(b.cfg.AI, logger)
    wssClient.Connect(ctx, connectParams)
    aiClient = wssClient
}
6 Backoffice Client (Esqueleto) Preparado para consultar tenants al CTN

Se ha creado la infraestructura para consultar la API de backoffice del CTN y resolver tenants por DDI. La llamada HTTP real se implementara cuando el CTN entregue su spec de API.

Configuracion YAML

# En config.yaml (cuando se active)
backoffice:
  enabled: true
  url: "https://api.ctn.es/v1"        # URL del backoffice CTN
  api_key: "ctn_secret_key"          # Clave de autenticacion
  cache_ttl: 300                      # Cache TTL en segundos (default 5 min)

Arquitectura del Cache

Llamada entrante → DDI: 917374206 | v [BackofficeClient.GetByDDI("917374206")] | cache hit? ──si──> Devuelve TenantConfig del cache | no | HTTP GET {url}/tenants?ddi=917374206 (pendiente de spec) | Guarda en cache con TTL | v Devuelve TenantConfig

Flujo de fallback previsto

Cuando el backoffice este activo, el bridge seguira esta cadena de resolucion de tenant:

  1. tenants.LookupByID(notariaID) — config local
  2. Si falla: tenants.LookupByDDI(ddi) — config local
  3. Si falla y backoffice.enabled: backofficeClient.GetByDDI(ddi) — API remota + cache
  4. Si todo falla: fallback transfer al default

10 Backup y Rollback Como revertir si es necesario

Se creo un backup de los archivos originales antes del despliegue:

# Ubicacion del backup
/opt/audio-bridge-backups/pre-ctn-20260407/
  ├── bridge.go      # internal/bridge/bridge.go original
  ├── server.go      # internal/api/server.go original
  └── config.go      # internal/config/config.go original

Para hacer rollback:

# 1. Restaurar archivos
cp /opt/audio-bridge-backups/pre-ctn-20260407/bridge.go /opt/audio-bridge/internal/bridge/
cp /opt/audio-bridge-backups/pre-ctn-20260407/server.go /opt/audio-bridge/internal/api/
cp /opt/audio-bridge-backups/pre-ctn-20260407/config.go /opt/audio-bridge/internal/config/
rm /opt/audio-bridge/internal/config/backoffice_client.go

# 2. Recompilar
cd /opt/audio-bridge && go build -o audio-bridge cmd/bridge/main.go

# 3. Reiniciar servicio
systemctl restart audio-bridge
